## Agent 开发规范

仅供 Agent 阅读。

### 适用规模

- **大规模 Version**：需求点 ≥ 5，或存在明显可并行子任务、代码依赖复杂 —— 拆四阶段执行，SubAgent 并行开发。
- **小规模 Version**：需求点少、依赖简单 —— MainAgent 独立完成。

无论规模，都遵守：**问题清单驱动、零猜测、零上下文交付、新功能强制单元测试 + 启动测试**。

---

### 1. 核心原则

#### 1.1 Version 四阶段

每个 Version 的开发工作按顺序分为四个阶段：

| 阶段 | 产出物 | 目标 |
|------|--------|------|
| **阶段一：提问** | 问题清单.md（v1/v2/v3...） | 澄清所有不确定性 |
| **阶段二：设计** | 设计文档 + 开发文档 + 验收文档 | 零上下文可开发、可验收 |
| **阶段三：开发** | 可运行的代码 + 单元测试 + 启动测试 | 功能实现 |
| **阶段四：验收** | 验收报告（编译/启动/功能/整体） | 质量门禁 |

**阶段间门禁**：每个阶段完成后，MainAgent 将中间产物（该阶段的所有交付物）提交用户检查。**用户确认后，才能进入下一阶段**。

#### 1.2 问题清单驱动（核心）

**问题清单贯穿所有阶段**。每个阶段内部遇到的技术分歧、不确定性，都要写入该阶段的 `问题清单.md`，并按置信度分区：

- **置顶区**（阶段级别阻塞）：置信度 < 90% **且** 涉及重大技术路线分歧，或严重前期设计失误，**必须用户决策才能继续**的问题。置顶区未清空时，阶段不截断，但异常处理可暂停对应工作。
- **自决区**（不阻塞阶段流）：≥ 90% 置信且无路线分歧的问题。MainAgent 自主决策，但完整记录方案与自决理由。

**决策原则**：

- **不要因细枝末节截断阶段**。只有置信度 < 90% **且** 涉及重大技术路线分歧、严重影响后续设计/开发方向的问题，才放入置顶区请求用户判断。
- 一般性接口选择、实现细节（如变量命名、枚举值顺序、文件拆分粒度），即使置信度略低，也归入自决区，记录方案和理由后继续推进。
- **阶段内正常推进**：除非遇到"如果选错方向，后续大量返工"级别的问题，否则不阻塞。

**问题清单的迭代机制：**

1. **v1**：进入新阶段时，MainAgent 梳理当前阶段的不确定点，生成 `问题清单.md` 初版。
2. **等答复**：置顶区问题以独立回复发给用户。用户答复前不做该阶段实质工作。
3. **v2、v3 ...**：用户答复后若引出新问题，追加到清单末尾，标记 `## v2 追问`、`## v3 追问`。
4. **清空为止**：置顶区问题全部关闭后，进入阶段的实质工作。
5. **阶段中追加**：阶段进行中出现新不确定点，追加到当前阶段的问题清单，自决区自行处理，置顶区上报用户。

#### 1.3 零上下文交付

任何开发 / 验收文档，都要让一个全新 Agent 在没有对话历史的情况下能独立理解、执行、判断。设计文档尤其要为 SubAgent 优化（见第 6.1 节）。

---

### 2. 角色分工

| 角色 | 职责 |
|------|------|
| **MainAgent** | 主导四个阶段；维护问题清单；编写设计/开发/验收文档；分派 SubAgent；协调测试与验收；处理异常与重试；提交中间产物给用户检查。 |
| **SubAgent** | 按 SubStage 任务书开发，不扩边界；遇不确定立即停并报告 MainAgent；完成后自测并报告；不与其他 SubAgent 沟通。 |
| **验收 Agent** | 只依据验收文档 + 设计/开发文档 + 完成报告 + 仓库当前状态验收；不依赖对话历史；每项给出通过/失败/阻塞并附证据；文档不清则要求补充。 |

---

### 3. 开发流程（四阶段）

#### 3.0 版本规划入口

1. 确认需求并分配语义化版本号。
2. 确定适用规模（大规模 → 四阶段；小规模 → MainAgent 独立完成）。

#### 3.1 阶段一：提问

**目标**：澄清 Version 级别所有不确定性，确保后续设计建立在明确的需求基础上。

1. MainAgent 梳理需求，列出所有不确定点，创建 Version 级 `问题清单.md`（v1），按 1.2 分区。
2. **以独立回复把 v1 清单发给用户，等答复**。
3. 用户答复后若引出新问题，追加 `v2 追问`，继续独立回复等待，反复直到置顶区清空。
4. 置顶区清空后，产出最终 `问题清单.md`。

**产出物**：`问题清单.md`（所有问题已关闭或自决）

**阶段门禁**：提交用户检查问题清单，确认无误后进入阶段二。

#### 3.2 阶段二：设计

**目标**：产出零上下文的设计文档、开发文档、验收文档，新的 Agent 拿到后可直接开工。

1. **合并需求与问题清单**：将原始需求 + 阶段一问题清单的解答，转化为完整的方案设计。
2. **划分 Stage 和 SubStage**：
   - 按代码依赖关系排列 Stage 顺序（Stage 之间串行）。
   - 每个 Stage 内拆分为可并行的 SubStage（最小工作单元）。
   - 写清跨 SubStage 契约（数据格式、接口、路径、字段、生命周期、错误处理）。
3. **编写设计文档**：需满足 6.1 零上下文可开发标准，针对 SubAgent 优化（见 6.1.1）。
4. **编写开发文档（SubStage 任务书）**：每个 SubStage 一份，含完整接口契约、输入输出、验收标准、禁止事项、依赖前置。
5. **编写验收文档**：需满足 6.3 零上下文可验收标准。
6. MainAgent 自检（6.1 自检问题逐条回答），通过后方可定稿。

**产出物**：
- `设计文档.md`（总体架构、Stage 划分、跨 Stage 契约）
- `开发文档/`（每个 SubStage 一份任务书）
- `验收文档.md`

**阶段门禁**：提交用户检查所有文档，确认设计合理、文档零上下文可读后，进入阶段三。

#### 3.3 阶段三：开发

**目标**：充分调动 SubAgent 按任务书并行开发，完成编码、单元测试和初步验证。

1. **分发 SubStage**：按 SubStage 任务书将任务分发给空闲 SubAgent。可并行分发的尽量并行。
2. **SubAgent 开发**：
   - 严格按任务书实现，不扩边界。
   - 遇不确定性问题立即停并报告 MainAgent。MainAgent 按 1.2 规则处理（自决区自行决定，置顶区上报用户）。
3. **自测**：每个 SubAgent 按任务书执行构建、单元测试。
4. **汇总集成**：MainAgent 处理冲突、接口偏差、文档同步。
5. **启动测试**：执行启动测试（见第 4 节），验证应用无崩溃。
6. 任意 SubStage 失败最多重试 3 次，仍失败则暂停并重规划。

**产出物**：
- 可运行的代码
- 单元测试结果
- 启动测试证据

**阶段门禁**：提交用户检查代码运行状态、测试结果，确认后可进入阶段四。

#### 3.4 阶段四：验收

**目标**：从编译、启动、功能完整性、整体四个层次逐级验收。

验收工作按以下四个层次依次执行：

1. **编译测试**：全量构建通过，无编译错误/警告。
2. **启动测试**：应用无崩溃启动，主窗口正常显示，新功能入口可触发。
3. **阶段功能完整性测试**：按验收文档逐一验证每个验收项，记录通过/失败/阻塞与证据。
4. **整体测试**：全量系统测试 + 回归测试，确保历史功能不受影响。

失败处理：
- 编译失败 → 退回阶段三修复。
- 启动测试失败 → 退回阶段三修复。
- 功能验收失败 → 定位到 SubStage 修复；验收标准不清则先补文档再验收。
- 整体测试失败 → 定位异常项修复。

**产出物**：
- `验收报告.md`（按四个层次记录测试结果、通过/失败项及证据）

**阶段门禁**：提交用户检查验收报告，所有验收项通过后 Version 正式完成。

#### 3.5 Version 收尾

- 产出 Version 总结报告：阶段汇总、测试结果、验收结论、版本号、变更日志、所有问题清单归档。
- 检查所有阶段的问题清单确保无未关闭项。
- 打 tag（如 `v2.1.0`）。

---

### 4. 测试与验收

**新功能 / 较大更新**（满足任一即算）：

- 新增功能模块、对话框、数据流、协议实现、Modbus 寄存器路径。
- 新增文件 ≥ 3 个 或 影响文件 ≥ 5 个。
- 涉及信号槽新连接、跨线程通信、OpenGL/QChart/QML 渲染改动。
- 触及"Critical Recurring Bugs"任一模块。

**强制**：单元测试 + 启动测试，缺一不可。不允许仅凭编译通过就报告完成。

- **单元测试**：新增 / 修改核心逻辑写 gtest / QtTest，覆盖正常路径 + 至少一个边界 / 异常，实际执行并通过。
- **启动测试**：实际启动 `TactileSense.exe`，走一遍新功能核心路径，验证无崩溃、无异常日志、UI 无异常。证据（截图说明、日志片段、操作记录）写入完成报告。

**小规模修改**（≤ 2 文件、纯 bug fix、无新增数据流）可豁免启动测试，但既有测试须跑通。

| 类型 | 触发 | 范围 | 输出 |
|------|------|------|------|
| 编译测试 | 阶段四 | 全量构建 | 编译日志 |
| 启动测试 | 阶段三结束 + 阶段四 | 本机启动 + 核心路径 | 截图 / 日志 / 操作记录 |
| 功能完整性测试 | 阶段四 | 按验收文档逐项 | 验收记录 |
| 系统测试（全量） | 阶段四 | 整个系统 | 测试报告 |

**启动测试通过标准**：

- 应用无崩溃启动，主窗口正常显示。
- 新功能入口可触发，进入后无立即崩溃。
- 核心操作路径至少完整走一遍无异常。
- `qDebug/qWarning/qCritical` 无与改动相关报错；spdlog 无 ERROR。
- 无法本机启动（缺硬件、需真实设备）：完成报告写明阻塞原因，并追加到 `问题清单.md` 置顶区。

---

### 5. 异常处理

- **SubStage 失败**：MainAgent 通知重试，最多 3 次；重试前可要求分析原因。
- **3 次失败**：暂停阶段，重新拆分 SubStage 或重写方案。
- **开发中不确定**：追加当前阶段的问题清单，按 1.2 处理。
- **验收失败**：定位 SubStage 修复；验收标准不清则先修文档再验收。
- **不允许**：
  - SubAgent 擅改任务边界。
  - 验收者用对话历史代替验收文档。
  - 无法执行的验收项默认通过。
  - 跳过启动测试就报告新功能完成。
  - **因非重大分歧截断阶段**（见 1.2 决策原则）。

### 6. 文档标准

| 工件 | 时机 | 说明 |
|------|------|------|
| Version 问题清单 | 阶段一 | 按 6.2 分区，含 v1/v2 迭代记录 |
| 阶段二问题清单 | 阶段二进行中 | 设计阶段特有的不确定点 |
| 阶段三问题清单 | 阶段三进行中 | 开发阶段特有的不确定点 |
| 设计文档 | 阶段二 | 满足 6.1，含 Stage/SubStage 划分 |
| 开发文档（SubStage 任务书） | 阶段二 | 每个 SubStage 一份，满足 6.1 |
| 验收文档 | 阶段二 | 满足 6.3 |
| 验收报告 | 阶段四 | 四个层次的测试结果与证据 |
| Version 总结报告 | Version 收尾 | Stage 汇总、全量测试、验收结论、版本号、变更日志、问题清单归档 |

#### 6.1 零上下文可开发

设计文档 / SubStage 任务书要包含：

1. **背景与目标**（1-3 句，不用"如之前讨论的"等引用）。
2. **当前代码状态**（相对路径 + 必要行号）。
3. **完整接口契约**（数据结构、枚举、常量、字段名、生命周期、错误处理）。
4. **输入输出**（来源、去向、格式）。
5. **验收标准**（可测试、可观察）。
6. **禁止事项**（不该改的模块 / 文件）。
7. **依赖前置**（依赖产物的位置、格式、可用条件）。

自检：新 Agent 读完能否直接开工？是否有依赖对话历史的表述？接口是否可直接写代码？

##### 6.1.1 文档对 SubAgent 的优化要求

设计文档和开发文档（SubStage 任务书）必须针对 SubAgent 的工作模式优化：

1. **任务自足**：每个 SubStage 任务书必须是独立的——SubAgent 仅凭自己的任务书就能完成工作，不需要阅读其他 SubStage 的任务书。
2. **跨 SubStage 契约明确**：如果 SubStage A 的产出被 SubStage B 使用，必须在两个任务书中都明确写出该契约（数据结构、文件路径、函数签名），而不是让 SubAgent 互相沟通。
3. **流程控制点**：
   - 明确标注"等待依赖"：如果本 SubStage 依赖前置工作，写明"需等待 XXX 完成后，在 YYY 路径获取 ZZZ 文件/数据结构"。
   - 明确标注"独立可执行"：不依赖其他 SubStage 的 SubStage，标注"无需等待任何前置，可立即开工"。
4. **阶段约束说明**：
   - 在每个 SubStage 任务书开头用固定标记说明所属阶段和依赖关系：
     ```markdown
     ## SubStage: [名称]
     - 所属 Stage：Stage N
     - 依赖前置：SubStage A / 无依赖
     - 并行状态：[可与其他 SubStage 并行 / 需等待]
     - 所属阶段：阶段三 - 开发
     ```
5. **错误处理契约**：明确写明接口/模块在出错时的行为——返回错误码还是抛出异常、错误信息格式、调用方如何处理。

#### 6.2 问题清单标准

**每条问题包含：**

1. **问题描述**：定位到具体文件 / 模块 / 行号。
2. **建议方案 / 说明**：基于代码与上下文给出推荐答案。
   - 优先鲁棒性和可测试性。
   - 解决实际问题，不加无用抽象。
   - 不以"简单快速""开发成本低"作为理由（开发由 Agent 执行，工期不是约束）。
3. **风险预警**：无法由 Agent 规避的风险标注"需用户决策"。
4. **备选方案**：多条合理路径或推荐方案风险高时，给 1-2 个备选并对比。
5. **置信度**：百分比或高 / 中 / 低。
6. **分区归属**：
   - ≥ 90% 且无路线分歧 → **自决区**，写自决理由。
   - < 90% **且** 涉及重大技术路线分歧 → **置顶区**，阻塞开发。
   - < 90% **但不涉及重大分歧** → 仍归自决区，记录方案后继续推进，由用户后续检查时确认。

**推荐结构：**

```markdown
# 问题清单 - vX.Y.Z / 阶段X

## v1（初次梳理，YYYY-MM-DD）

### 置顶区（待用户确认，阶段阻塞）

#### Q1. 问题标题
- 问题描述：...
- 建议方案：...
- 备选方案：...
- 风险预警：...
- 置信度：70%
- 归入置顶原因：例如"两种方案对后续扩展性差异显著，需用户拍板"。

### 自决区（≥ 90% 置信，或 < 90% 但不涉及重大分歧，不阻塞）

#### Q2. 问题标题
- 问题描述：...
- 自决方案：...
- 自决理由：例如"Python reference `src/device_center/protocol.py:123` 已有明确实现"。
- 风险预警：...
- 置信度：95%

## v2 追问（用户答复 v1 后新引出的问题，YYYY-MM-DD）

### 置顶区
...
### 自决区
...
```

**迭代机制：**

- 每次生成或追加清单后，**以独立回复发送给用户**，正文只放清单，不夹带其他工作。
- 用户答复前不做该阶段含置顶区问题的实质工作。
- 用户答复后若引出新问题，追加 `## v2 追问`、`## v3 追问`，继续独立回复等待。
- **置顶区所有问题被答复关闭后**，该阶段的实质工作方可继续。
- 自决区问题不阻塞阶段流，MainAgent 可继续推进。

**自检：**

- 每个问题都有建议方案和置信度吗？
- 置顶区是否只包含真正需要用户决策的重大问题？
- 是否有"虽置信度 < 90% 但不涉及路线分歧"的问题被错误放入置顶区？
- 自决区的自决理由是否充分（避免用虚高置信度绕过用户）？
- 用户仅凭置顶区能否作"同意 / 否决 / 选备选"的判断？

**不允许：**

- 无建议的裸问题。
- 多个独立问题塞一条。
- 建议里省略风险。
- 用"赶工""简单"作为核心理由。
- 不写入清单直接开发（100% 有把握也要落自决区）。
- 明显 ≥ 90% 且无分歧的问题塞置顶区打扰用户；反之，有分歧或低置信度问题塞自决区跳过用户决策。
- 因非重大分歧截断阶段流。

#### 6.3 零上下文可验收

验收文档要包含：

1. **验收范围**（覆盖 / 不覆盖）。
2. **验收环境**（系统、构建配置、测试数据、硬件、外部服务、账号）。
3. **前置条件**（仓库、配置、数据、设备初始状态）。
4. **验收项列表**（唯一编号、目的、操作步骤、期望结果、通过 / 失败标准）。
5. **证据要求**（命令输出、文件路径、日志片段、截图说明、人工记录）。
6. **失败与阻塞处理**（何时记录、重试、追加问题清单、退回开发）。
7. **回归范围**（历史功能哪些回归、哪些冒烟、哪些不覆盖）。
8. **交付物检查**（必须存在的文档、报告、配置、样例数据）。

**验收项格式：**

```markdown
### A1. 验收项标题
- 目的：...
- 前置条件：...
- 操作步骤：...
- 期望结果：...
- 通过标准：...
- 失败标准：...
- 证据要求：...
```

**不允许：**

- "按之前讨论"、"看起来正常"等主观表述。
- 只写命令不写期望输出与失败处理。
- 无法执行的硬件 / 人工验收项默认通过。
- 开发完成后才补验收标准。

---

### 7. 强制门禁

以下任一违反，都不能标记 Stage / Version 完成：

- **置顶区未清空**：问题清单里还有待用户确认的重大问题。
- **验收标准不清**：验收文档缺项或存在主观表述。
- **新功能未跑启动测试**：或未在验收报告中提供启动测试证据 / 豁免说明。
- **阶段间未提交用户检查**：未完成用户确认即进入下一阶段。
- **因非重大分歧截断阶段**：将可自决的问题提交用户决策，导致不必要的中断。
